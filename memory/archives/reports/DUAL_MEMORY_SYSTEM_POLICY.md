## 1. 背景

当前存在两套记忆系统：

| 系统 | 位置 | 用途 | 实体数 |
|------|------|------|--------|
| **KG** (Knowledge Graph) | `memory/ontology/graph.jsonl` | 项目/技能知识 | 3,097 |
| **XMO** | `~/.xmo/kg/entities.jsonl` | 个人/会话记忆 | 3,389 |

为避免混淆、明确分工，特制定本规范。

## 2. 分工定义

### 2.1 KG 存储范围

| 类别 | 示例 |
|------|------|
| **技能配置** | skill_tracker_asset.json, subagents-config.yaml |
| **工作流定义** | CronJob 定义、自动化流程 |
| **项目进度** | Milestone, Progress, 阶段性成果 |
| **技术架构** | Schema、API设计、系统配置 |
| **工具/平台集成** | Feishu、GitHub、Moltbook 配置记录 |
| **可复用知识** | 解决方案、技术文档 |

### 2.2 XMO 存储范围

| 类别 | 示例 |
|------|------|
| **用户偏好** | 称呼方式、语言、时区 |
| **个人承诺** | 心炙的承诺、协作保证 |
| **重要时刻** | 第一个拥抱、第一个语音 |
| **协作原则** | 沟通原则、决策流程 |
| **个人教训** | Bounty 失败教训、错误总结 |
| **会话决策** | 重要对话中的关键决定 |
| **情感记忆** | 有意义的互动、里程碑 |

## 3. 写入规范

### 3.1 判断流程

```
新记忆 → 判断类型 → 选择系统 → 记录

判断标准：
├── 这是一般性知识/技术信息吗？ → KG
├── 这是否与特定项目/技能相关？ → KG
├── 这是否可复用/可版本化？ → KG
│
├── 这是个人偏好或主观决策吗？ → XMO
├── 这需要跨平台访问吗？ → XMO
├── 这是情感/关系相关的记忆吗？ → XMO
```

### 3.2 常用命令

**写入 KG：**
```bash
# 通过 memory-optimization 工具
python3 memory_ontology.py create --type Decision --props '{"title":"...","rationale":"..."}'
```

**写入 XMO：**
```bash
# 通过 MCP 工具
xmo_extract --type Finding --name "重要发现" --content "..."
```

### 3.3 双写规则（特殊场景）

以下情况需要**同时写入两个系统**：

| 场景 | KG | XMO |
|------|-----|-----|
| **重要技术决策** | ✅ 完整记录 | ✅ 摘要 |
| **项目里程碑** | ✅ 详细进度 | ✅ 关键要点 |
| **技能配置变更** | ✅ 完整配置 | ✅ 变更记录 |

**示例：技能配置更新**
```bash
# 1. 写入 KG（完整配置）
python3 memory_ontology.py create --type Skill --props '{
  "name": "new-skill",
  "config": {...},
  "version": "1.0"
}'

# 2. 写入 XMO（变更记录）
xmo_extract --type Finding --name "新技能安装" --content "安装 new-skill v1.0" --tags "skill,change"
```

## 4. 查询规范

### 4.1 查询场景

| 场景 | 推荐系统 | 原因 |
|------|----------|------|
| 查技能配置 | KG | 结构化、完整 |
| 查项目进度 | KG | 详细、时间线 |
| 查技术方案 | KG | 可复用 |
| 查用户偏好 | XMO | 快速定位 |
| 查个人承诺 | XMO | 完整记录 |
| 查历史教训 | XMO | 按时间排序 |
| 查重要时刻 | XMO | 情感记忆 |

### 4.2 查询命令

**KG 查询：**
```bash
# 搜索实体
python3 memory_ontology.py query --type Decision

# 查看统计
python3 memory_ontology.py stats

# 验证图谱
python3 memory_ontology.py validate
```

**XMO 查询：**
```bash
# 语义搜索
xmo_query --keywords "偏好,rong"

# 查看统计
xmo_stats
```

### 4.3 聚合查询

复杂问题需要同时查两个系统：
```bash
# 分別查询后合并结果
echo "=== KG 结果 ==="
python3 memory_ontology.py query --type Skill

echo "=== XMO 结果 ==="
xmo_query --keywords "skill"
```

## 5. 冲突解决

### 5.1 数据不一致

| 情况 | 处理方式 |
|------|----------|
| 同实体两系统都有 | 保留更新时间最新的 |
| 内容冲突 | KG 优先（结构化更可靠） |
| 重复实体 | 标记并逐步清理 |

### 5.2 写入冲突

| 情况 | 处理方式 |
|------|----------|
| 写入失败 | 记录到待重试队列 |
| 锁竞争 | 等待 30 秒后重试 |

## 7. 监控与告警

### 7.1 健康检查

| 指标 | 阈值 | 检查频率 |
|------|------|----------|
| KG 实体数 | > 1000 | 每日 |
| XMO 实体数 | > 1000 | 每日 |
| 最后写入时间 | < 7 天 | 每日 |
| 图谱验证 | 100% 通过 | 每周 |

### 7.2 告警规则

- ❌ 任一系统 7 天无写入
- ❌ 图谱验证失败
- ❌ 实体数异常下降 > 50%

## 8. 审查机制

### 8.1 每周检查清单

```
□ 检查过去 7 天新实体是否在正确系统
□ 检查是否有遗漏的重要记忆
□ 验证双写场景是否执行
□ 查看是否有写入失败记录
```

### 8.2 每月回顾

- 评估分工是否合理
- 统计两系统使用频率
- 识别可优化点
- 更新本规范（如有必要）
